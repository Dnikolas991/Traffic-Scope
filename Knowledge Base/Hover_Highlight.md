# Cities: Skylines II Hover / Highlight Notes

This document captures the currently confirmed vanilla hover/highlight behavior and the implementation constraints it creates for Transit Scope.

## 1. Core Distinction

### `Highlighted + Updated` is not the full hover preview path

Confirmed:

- `Game.Tools.Highlighted` is consumed by rendering systems.
- It reliably drives outline/highlight behavior.
- It does not by itself produce the full vanilla hover preview.

Implication:

- Adding `Highlighted + Updated` to the original entity can produce blue outlines or partial highlight.
- It cannot reproduce vanilla road hover mask.
- It also cannot reproduce vanilla building hover coverage across the full composition.

## 2. Vanilla Hover Lifecycle

Vanilla hover preview depends on a full tool lifecycle:

1. The active tool refreshes `CreationDefinition`.
2. Downstream systems generate temp/helper preview entities.
3. Rendering systems consume those temp/helper entities.
4. Apply/cleanup systems replace, reuse, and clear old preview state.

Implication:

- A single reflected method call is usually insufficient.
- Partial helper-definition simulation is unsafe.
- A few copied tool state fields are not enough to become equivalent to vanilla hover.

## 3. Road Hover

### Confirmed path

For ordinary default-tool road hover, the entry chain is:

- `Game.Tools.ToolSystem::ToolUpdate()`
- `Game.Tools.DefaultToolSystem::OnUpdate()`
- `Game.Tools.DefaultToolSystem::Update(JobHandle)`

It is not `NetToolSystem`.

### Why outline appears without the vanilla mask

Confirmed:

- `Highlighted + Updated` reliably reaches `MeshLayer.Outline`.
- The light-blue road mask comes from temp preview net entities generated by:
  - `GenerateEdgesSystem`
  - `GenerateNodesSystem`
- The mask is not a simple overlay on the original edge.

Implication:

- ‚ÄúBlue outline only‚Äù is the expected result of the `Highlighted` route.
- The missing mask is not just a missing color constant.

### Why `UpdateDefinitions(...)` alone is insufficient

Confirmed:

- `DefaultToolSystem.UpdateDefinitions(...)` refreshes definitions only.
- It does not guarantee the full preview lifecycle.
- Stable replacement of old road preview also depends on:
  - `GenerateEdgesSystem`
  - `GenerateNodesSystem`
  - `ApplyNetSystem`

So these routes are known to be insufficient:

- calling only `UpdateDefinitions(...)`
- syncing only `m_LastRaycastEntity`
- syncing only `m_LastSelectedIndex`
- syncing only `applyMode`

### Current Transit Scope road baseline

Current stable project behavior:

- road hover is still owned by the mod tool
- the road source entity gets `Highlighted + Updated`
- `OverlaySystem` still draws the custom road hover overlay

This is stable and usable, but it is not vanilla-equivalent.

## 4. Building Hover

### Confirmed path

Default-tool building hover uses the same entry chain:

- `Game.Tools.ToolSystem::ToolUpdate()`
- `Game.Tools.DefaultToolSystem::OnUpdate()`
- `Game.Tools.DefaultToolSystem::Update(JobHandle)`

`DefaultToolSystem.UpdateDefinitions(...)` refreshes helper definitions, but does not directly render the final building hover visual.

### Building hover is not ‚Äúhighlight the hit entity‚Äù

Confirmed upward canonicalization relations:

- `Game.Tools.Temp.m_Original`
- `Game.Common.Owner.m_Owner`
- `Game.Objects.Attachment.m_Attached`
- `Game.Common.Target.m_Target`

Confirmed downward expansion relations:

- `Game.Objects.SubObject`
- `Game.Buildings.InstalledUpgrade`
- `Game.Objects.Attachment`
- `Game.Objects.Attached`

Implication:

- vanilla building hover first normalizes upward
- then expands back downward across the building composition

### Why relation walking is still not enough

Even if the relation set is correct, these pieces are still missing from a manual `Highlighted`-only implementation:

- real vanilla canonicalization order
- full `CreateDefinitionsJob::AddEntity(...)` output structure
- `GenerateObjectsSystem` temp/helper object creation
- `ApplyObjectsSystem` replacement and cleanup cadence

So ‚Äúthe root lights up, but floor/grass/attachments do not‚Äù is expected on the manual highlight route.

### Current Transit Scope building baseline

Current stable project behavior:

- the mod owns selection mode
- the mod resolves building hits manually
- the mod expands the building entity set manually
- the mod applies `Highlighted + Updated` to that original-entity set

This is stable enough to use, but it is not a full vanilla building hover reproduction.

## 5. Known Bad Routes

These routes have been proven insufficient or risky:

- only adding `Highlighted + Updated`
- only extending owner/target/subobject relation walking
- creating simplified helper definitions
- reflecting `UpdateDefinitions(...)` in isolation
- copying only a few private `DefaultToolSystem` fields

## 6. Recommended Constraint For Future Work

If hover is revisited later, the order of preference should be:

1. Let the real vanilla tool path run end-to-end.
2. If that is impossible, reconstruct the full definition input that downstream vanilla systems expect.
3. Do not use half-simulated helper-definition paths.
